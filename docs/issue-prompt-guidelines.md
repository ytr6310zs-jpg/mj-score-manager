**Issue対応プロンプトガイド**

- **目的:** 新しい Issue に対してエージェント（あなたや自動化エージェント）が一貫して効率的に動けるよう、推奨プロンプトとテンプレートを提供します。

**使い方（概略）**
- まず `Clarify` テンプレートで不明点を質問し、次に `Plan` で作業の分割と優先度を定義、`Implement` で実装指示、最後に `PR` 用テンプレートで成果物を整理します。

**推奨プロンプト構造**
- コンテキスト: リポジトリ名・該当ファイル・Issue 番号・目標
- 制約条件: CI 要求、互換性、ブランチ/コミットルール
- 出力物: 変更ファイル、テスト手順、PR本文、必要な確認項目
- 優先度/期限: もしあれば明記

---
テンプレート例（日本語）

1) Clarify（確認用）

```
Issue #<番号> を対応します。私に教えてください: 1) 目的の簡潔な要約、2) 既存の期待挙動、3) 優先度と期限、4) CI/運用上の制約。もし不明点があれば最大5つの具体的な確認質問を挙げてください。
```

2) Plan（設計）

```
この Issue を解決する短期的な実装計画を 3〜6 ステップで作ってください。各ステップに成果物と検証方法（コマンドや手順）を添えてください。ブランチ命名規則は `feature/issue-<番号>-<要約>`、ビルドは `npm run build` を必須とします。
```

3) Implement（実装指示）

```
上の Plan に従い、変更ファイルの最小差分パッチを準備してください。変更後に `npm run build` を実行し、成功したらコミット・push する手順まで含めてください。PR 本文に `spec.md` の有無とビルド結果を明記してください。
```

4) PR 本文テンプレ（短く）

```
設計概要: <要約>
実装概要: <追加/変更ファイルの簡潔リスト>
検証: `npm run build` 実行結果（成功/失敗）と手動確認手順
関連 Issue: #<番号>
```

---
注記:
- 機密情報はプロンプトやドキュメントに含めないでください。
- 自動変更を行う場合は、必ず `npm run build` を実行してからコミットしてください（ローカルポリシー）。
