# Copilot エージェント委任ガイド

このファイルは、Copilot エージェント（GitHub Copilot Chat）へ作業を委任するときの標準テンプレートと運用方針をまとめた参照ドキュメントです。

## 標準実行フロー

作業は **設計フェーズ** と **実装フェーズ** の2段階で進めます。

```
【設計フェーズ】
Issue読み取り → 設計資料作成（design/配下） → エージェント自己レビュー → ★ユーザー確認（必須停止）

【実装フェーズ（ユーザー承認後）】
実装 → テスト → 動作確認 → npm run build → コミット → push → PR作成
```

途中で失敗した場合は原因を解消してから次工程へ進み、未解消のままコミット・push しないこと。

---

## 委任テンプレート（コピー用）

### 設計フェーズの依頼

Issue に設計概要を記載した後、以下を貼ってエージェントに設計資料を作成させます。
エージェントは設計資料を作成・自己レビューした後、必ず確認のために停止します。

```
Issue #<番号> の概要に基づいて設計資料を作成してください。
設計資料は design/ 配下に作成し、自己レビュー結果もあわせて提示してください。
設計フェーズ完了後は私の確認を待ってください。
```

### 実装フェーズの依頼（設計承認後）

設計を確認・承認した後、以下を貼ると実装フェーズを止まらず完遂します。

```
設計を承認します。実装・テスト・動作確認・コミット・pushまで委任します。
軽微な実装判断は既存実装準拠で自動判断してください。
コード修正・関連テスト・必要最小限のドキュメント更新まで許可します。
npm run build は毎回実行し、可能なら npm run lint と npm test も実行してください。
コミットは自動で可、push も自動で可、PR作成も自動で可（ベースブランチ: develop）。
破壊的変更・機密情報関連・要件矛盾が発生した場合のみ確認して止まってください。
```

---

## 停止条件（エージェントが確認のために止まる場合）

| 条件 | フェーズ | 内容 |
|------|----------|------|
| 設計フェーズ完了 | 設計 | 設計資料の自己レビュー後、必ずユーザー確認を求める（必須） |
| 要件矛盾 | 両方 | 既存仕様と指示が明確に矛盾している |
| 破壊的変更 | 両方 | データ移行・スキーマ変更・公開I/F変更が伴う |
| 高リスク変更 | 両方 | 認証・認可・機密情報に関わる変更 |
| 環境不足 | 実装 | 外部権限・環境起因で検証が不可能 |
| ブランチ粒度の競合 | 実装 | 命名や粒度に複数の妥当案があり運用影響が大きい |

## 検証コマンド一覧

| コマンド | タイミング |
|----------|-----------|
| `npm run build` | コミット前（必須） |
| `npm run lint` | コミット前（可能なら必須） |
| `npm test` | コミット前（可能なら必須） |
| `npm run check:local-pages` | UI変更後（local Supabase 起動が必要） |

---

## PR本文に必須の記載項目

1. 設計概要（目的・対象・影響範囲）
2. 実装概要（変更内容の要約）
3. 実行した検証コマンドと結果
4. 手動または自動の動作確認内容
5. 既知の未解決事項
