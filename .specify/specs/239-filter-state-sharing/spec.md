# Feature Specification: フィルタ状態共有

**Feature Branch**: `feature/issue-239-filter-state-sharing`  
**Created**: 2026-05-08  
**Status**: Draft  
**Issue**: #239

---

## User Scenarios & Testing

### User Story 1 - フィルタページ間で同じ条件を引き継ぐ (Priority: P1)

ユーザーが対局履歴・成績集計・相性表のいずれかで大会と期間条件を設定したあと、別の集計ページへ移動しても同じ条件で画面を確認できる。

**Why this priority**: Issue の主目的そのものであり、同じ大会・同じ期間を跨いで比較できないと改善価値が出ない。

**Independent Test**: `/stats` で大会・期間・試合数条件を設定し、ヘッダーから `/matches` と `/compatibility` に移動して、各ページで有効な条件が引き継がれることを確認する。

**Acceptance Scenarios**:

1. **Given** ユーザーが成績集計で大会・開催日フィルタを変更した、**When** ヘッダーから対局履歴へ移動する、**Then** 対局履歴は同じ大会・同じ期間条件で表示される
2. **Given** ユーザーが相性表で大会・カスタム日付範囲・試合数条件を指定した、**When** ヘッダーから成績集計へ移動する、**Then** 成績集計は同じ大会・期間・試合数条件で表示される
3. **Given** ユーザーが対局履歴で大会と日付条件を指定した、**When** 相性表へ移動する、**Then** 大会と日付条件は引き継がれ、試合数条件は相性表側の既定動作に従う

---

### User Story 2 - スコア入力を挟んでもフィルタ状態を失わない (Priority: P1)

ユーザーがフィルタ済み画面を確認したあとにスコア入力へ移動し、再び対局履歴・成績集計・相性表へ戻っても直前のフィルタ条件を再利用できる。

**Why this priority**: Issue 本文で「スコア入力へ遷移しても保持したい」が明示されている。

**Independent Test**: フィルタ済みの `/stats` から `/` に移動し、その後 `/matches` へ移動して条件が復元されることを確認する。

**Acceptance Scenarios**:

1. **Given** ユーザーが成績集計で大会・日付条件を設定している、**When** スコア入力へ移動してから対局履歴へ戻る、**Then** 直前の大会・日付条件が復元される
2. **Given** ユーザーが相性表で試合数条件を設定している、**When** スコア入力へ移動してから成績集計へ戻る、**Then** 試合数条件を含む共有状態が復元される
3. **Given** スコア入力画面ではフィルタ UI を持たない、**When** 画面を表示する、**Then** 共有フィルタ状態は変更も削除もされない

---

### User Story 3 - 保存状態が使えない場合でも既存挙動を壊さない (Priority: P2)

保存済み状態が存在しない、壊れている、または現在のページで使えない場合でも、各画面は既存の初期表示ルールで動作する。

**Why this priority**: 永続化導入で初期表示や既存 URL 互換を壊すと運用影響が大きい。

**Independent Test**: localStorage を空または不正値にした状態で各画面を開き、既存の初期表示と互換 URL が維持されることを確認する。

**Acceptance Scenarios**:

1. **Given** 共有フィルタ状態が保存されていない、**When** `/matches` を直接開く、**Then** 既存どおり当年表示を初期値として描画される
2. **Given** 共有フィルタ状態の JSON が壊れている、**When** `/stats` を開く、**Then** 保存値は無視され、画面はエラーなく既定条件で表示される
3. **Given** URL に明示的なフィルタパラメータがある、**When** `/compatibility?filter=custom&start=2026-01-01&end=2026-01-31` を開く、**Then** URL パラメータが保存値より優先され、その値で表示される

---

## Edge Cases

- 保存状態に `tournamentId` があっても現在の大会一覧に存在しない場合は無視し、既存の大会初期選択へフォールバックする
- 保存状態に `minGames` があっても対局履歴では UI に表示せず、検索条件にも使わない
- 対局履歴から共有状態を更新した場合は `minGames` を消さず保持し、成績集計・相性表へ戻った際に再利用できるようにする
- `filter=custom` なのに `start/end` が欠けている保存状態は無効として扱う
- 旧 URL 互換の `mode/start/end` で開いた場合も最終的な正規化後の値を共有状態へ保存する

## Requirements

### 機能要件

| ID | 要件 |
|---|---|
| F-01 | 対局履歴・成績集計・相性表で共通利用するフィルタ状態をブラウザ保存できること |
| F-02 | 保存対象は `filter`, `start`, `end`, `tournamentId`, `minGames` とすること |
| F-03 | フィルタ UI 操作後は、現在ページで有効な条件を正規化して保存すること |
| F-04 | 対局履歴・成績集計・相性表を URL パラメータなしで開いた場合、保存済み状態が有効ならその条件を復元して表示すること |
| F-05 | URL パラメータが明示されている場合は URL を優先し、その値で表示したうえで保存状態を更新すること |
| F-06 | スコア入力画面へ移動しても共有フィルタ状態を保持し、スコア入力画面自身はその状態を破壊しないこと |
| F-07 | ヘッダー経由で対局履歴・成績集計・相性表を移動する際、保存済み条件があれば遷移先でも同じ条件を即時利用できること |
| F-08 | 対局履歴では `minGames` を無視し、成績集計・相性表では既存仕様どおり利用すること |
| F-09 | 大会選択の既存 localStorage 保持と矛盾しないこと。共有フィルタ保存時とスコア入力での大会変更時は大会選択も同じ値にそろえること |
| F-10 | 既存の `resolveFilterParams` による `filter` と旧 `mode/start/end` の互換解釈を維持すること |

### 非機能要件

| ID | 要件 |
|---|---|
| N-01 | 既存の Server Component ベースのページ構成を維持し、最小差分で実装すること |
| N-02 | 保存失敗や不正 JSON があってもページ表示を阻害しないこと |
| N-03 | ブラウザ保存はクライアントコンポーネントに閉じ、サーバーコードから `window` に依存しないこと |
| N-04 | 初回表示で保存状態を適用する処理は無限リダイレクトを起こさないこと |

## Design Decisions

### 1. 保存方式

大会保持と同じく `localStorage` を使う。理由は以下のとおり。

- 既存実装で大会選択が `localStorage` ベースで安定運用されている
- ブラウザをまたいだセッション共有は不要で、「前回選択状態保持」という Issue 意図に合う
- cookie や server session を追加せず最小差分で導入できる

### 2. URL と保存状態の優先順位

1. 明示された URL パラメータ
2. 保存済み共有フィルタ状態
3. 各ページの既存デフォルト

URL パラメータのみで保持する実装は採らないが、Server Component が検索条件を確定するため、保存状態から URL へ正規化して反映するクライアントブリッジを導入する。

### 2.1 URL を「明示指定」とみなす条件

以下のいずれかが存在する場合、そのページは URL を source of truth とみなす。

- 全フィルタページ共通: `filter`, `mode`, `start`, `end`, `tournamentId`
- 成績集計・相性表のみ: `minGames`

補足:

- `mode` は旧 URL 互換のため explicit 判定に含める
- `minGames` 単独指定でも、成績集計・相性表では explicit とみなす
- 対局履歴では `minGames` 単独指定は explicit 判定に含めない

### 3. ページ別の扱い

| ページ | filter/start/end | tournamentId | minGames |
|---|---|---|---|
| 対局履歴 | 使用する | 使用する | 保存はするが適用しない |
| 成績集計 | 使用する | 使用する | 使用する |
| 相性表 | 使用する | 使用する | 使用する |
| スコア入力 | UI では扱わない | 既存どおり使用する | 扱わない |

### 4. スコア入力画面での大会変更ルール

スコア入力画面で大会を変更した場合は、既存の大会保持キーに加えて共有フィルタ状態内の `tournamentId` も同じ値へ同期する。  
このとき `filter`, `start`, `end`, `minGames` は保持し、スコア入力画面はフィルタ状態全体を初期化しない。

理由:

- 大会選択は画面横断の文脈として扱う方が Issue の期待に合う
- スコア入力だけ別大会を選ぶと、一覧画面復帰時に大会だけ食い違う状態を避けられる

---

## Success Criteria

1. 対局履歴・成績集計・相性表の相互遷移で、大会と期間条件が保持されること
2. 成績集計・相性表の相互遷移では、`minGames` 条件も保持されること
3. 対局履歴では `minGames` を適用せず、成績集計・相性表へ戻ったときは保存済み `minGames` を再利用できること
4. スコア入力を挟んでも保存状態が失われず、スコア入力で大会を変更した場合は共有状態の `tournamentId` も更新されること
5. 保存値がない場合、不正 JSON の場合、旧 URL 互換の `mode/start/end` の場合でも画面が破綻しないこと
6. URL 明示指定がある場合、保存値ではなく URL が優先されること
