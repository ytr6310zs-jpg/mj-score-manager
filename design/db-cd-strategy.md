# DB CD Strategy (本番マイグレーション運用戦略)

目的
- 本番環境で安全かつ可観測にスキーマ変更を適用するための手順と責任範囲を定義する。

前提条件
- マイグレーションは `node-pg-migrate` でバージョン管理する。
- すべてのマイグレーションはステージングで検証済みであること。
- 本番は定期的なバックアップ（ポイントインタイム／スナップショット）が取得可能であること。
- ロールバック戦略はマイグレーションの内容によって手順が異なる（後述）。

高レベル手順
1. バックアップ取得
   - フルスナップショットとWALアーカイブ（可能ならPITR）を取得する。
   - 取得が完了・検証できることを確認する。

2. メンテナンス準備（必要に応じて）
   - 大幅なスキーマ変更（列削除・型変更など）はメンテナンスウィンドウで実行。
   - 影響範囲が小さければローリング方式で実施可能。

3. 実行者と権限
   - 実行は運用担当またはデプロイ権限を持つ開発者が行う。
   - 実行者は事前にチェックリストを確認し、Slack/Teams等で通知する。

4. マイグレーション適用
   - 推奨: `node-pg-migrate up` を実行する。CIによる自動実行はリスクが高いので手動トリガー推奨（ただし自動化許可ポリシーを整備した場合は別）。
   - 実行コマンド（例）:
     - `DATABASE_URL="$(cat /run/secrets/DB_URL)" node-pg-migrate up`

5. 検証（ポストマイグレーション）
   - サービスのヘルスチェックを実行（例: /health エンドポイント）。
   - 重要なクエリの速度・エラー率を監視（APM/ログ）。
   - データ整合性のサンプリング検証（主要テーブルの行数、ユニーク制約の検査など）。

6. ロールアウト完了報告
   - 問題がなければステータスを更新し、関係者に通知。

ロールバック方針
- ロールバック可能な変更: 非破壊的変更（列追加、インデックス追加等）は簡単に巻き戻せる。
- ロールバック困難な変更: 列削除・データ変換・構造的変更は基本的にロールフォワード（追加のマイグレーションで修正）を推奨。
- ロールバック手順（破壊的でない場合の例）:
  1. 事前バックアップを復元して検証環境で復旧リハーサルを行う。
  2. `node-pg-migrate down` を用いて戻す（ただし down スクリプトが安全であることを事前確認すること）。

監視とアラート
- マイグレーション直後 30 分はエラーレートとレイテンシを厳しく監視。
- 異常が検出されたら即時ロールバックまたはロールフォワード計画を開始。

運用チェックリスト（実行時）
- バックアップが成功している
- ステージングで同一マイグレーションが通っている
- 監視・アラートが設定されている
- 実行者が連絡先と手順を把握している

GitHub Actions（参考ワークフロー片段）
```yaml
name: migrate-prod
on:
  workflow_dispatch:
jobs:
  migrate:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4
      - name: Install
        run: npm ci
      - name: Run migrations (manual trigger)
        env:
          DATABASE_URL: ${{ secrets.PROD_DATABASE_URL }}
        run: |
          node -e "const {spawnSync} = require('child_process'); const env = Object.assign({}, process.env); const r = spawnSync('node-pg-migrate',['up'],{stdio:'inherit', env}); process.exit(r.status||0);"
```

備考・制約
- 本番マイグレーションは慎重に扱う。重大な破壊的変更は段階的に行い、互換性レイヤを用いること。
- down スクリプトは常に簡単に戻せることを保証するのではなく、ダウンタイムやデータ復旧計画が必要になる可能性がある点を明記する。

次のアクション候補
- このドキュメントを基に GitHub Actions の `migrate-prod` ジョブを作成する。
- ステージング用の同一ワークフローを作り、レビュー後に本番用シークレットを設定する。
