# Issue #157 設計: 成績集計画面 PDF出力機能

- Issue: https://github.com/ytr6310zs-jpg/mj-score-manager/issues/157
- タイトル: 成績集計画面 PDF出力機能
- 要件: 成績集計画面のフィルタ結果（開催日/今年/任意 + 試合数条件）を A4 横向き PDF として出力する（複数ページ可）

## 1. 目的

開催日の結果を後日共有しやすくするため、成績集計画面で絞り込んだ結果をそのまま PDF で配布できるようにする。

## 2. 現状

- 成績集計画面は [app/stats/page.tsx](app/stats/page.tsx) で表示している。
- 対局履歴データは [lib/matches.ts](lib/matches.ts) の `fetchMatchResults` で取得できる。
- 成績集計データは [lib/stats.ts](lib/stats.ts) の `fetchPlayerStats` で取得できる。
- エクスポート機能は CSV のみ存在し、PDF 出力は未実装。
  - [components/csv-export-button.tsx](components/csv-export-button.tsx)
  - [app/api/export/stats/route.ts](app/api/export/stats/route.ts)

## 3. 要件整理

Issue #157 から、以下を必須要件とする。

1. 出力対象は「現在のフィルタ条件で絞り込まれた対局履歴」と「同条件の成績集計」。
2. PDF は A4 横向き。
3. データ量が多い場合は複数ページでよい。

追加で実装時に必要な運用要件を以下とする。

1. システム日付を対象日の基準として使わない（対象期間はフィルタ値のみで決定する）。
2. フィルタ条件を PDF 内ヘッダーに明記する。
3. 画面の表示条件と PDF の集計条件を一致させる。
4. 日本語が文字化けしないこと。

## 4. 設計方針

## 4.1 方針A（採用）: 印刷用ページ + ブラウザの PDF 保存

初期実装は、印刷専用ページを生成し `window.print()` を起動する方式を採用する。

採用理由:

1. 追加ライブラリなしで実装できる。
2. A4 横向き、改ページ制御を CSS (`@page`, `page-break-*`) で実現できる。
3. ブラウザ描画ベースのため日本語フォントの扱いが安定しやすい。

## 4.2 方針B（非採用）: サーバーで PDF バイナリを直接生成

`pdf-lib` や `pdfkit` による API 生成は今回は採用しない。

非採用理由:

1. 日本語フォント埋め込みと表組みの実装コストが高い。
2. サーバー実行環境差分によりレイアウト崩れ調査コストが増える。

## 4.3 出力導線

1. 成績集計画面に「PDF 出力」ボタンを追加する。
2. ボタンクリックで印刷用ページに遷移する。
3. 印刷用ページはロード完了後に印刷ダイアログを開く。
4. ユーザーが「PDF に保存」を選択して保存する。

## 5. 画面/データ仕様

## 5.1 印刷対象ページ

新規ページ案:

- [app/stats/print/page.tsx](app/stats/print/page.tsx)

表示ブロック（4ブロック構成）:

1. レポートヘッダー — タイトル、対象期間、試合数条件、フィルタ要約。
2. 成績集計（フィルタ対象） — 画面と同一の主要列（順位、名前、対局数、合計点、主要率）。
3. 参考集計 + 別表 — 「今年の集計（20試合以上）」などの参考統計を別ブロックで出力し、続けて画面の別表（追加統計）を同順で出力する。出力順は画面と一致させる。
4. 対局履歴（1試合1行・圧縮表示） — 1試合につき1行で以下の列を表示する。

  | 列 | 内容 | 備考 |
  |---|---|---|
  | 対局日 | YYYY-MM-DD | |
  | 形式 | 3P / 4P | |
  | 1位 | プレイヤー名 | |
  | 2位 | プレイヤー名 | |
  | 3位 | プレイヤー名 | |
  | 4位 | プレイヤー名 | 3人打ち時は列ごと非表示 |
  | 飛ばし | 飛ばしプレイヤー名（複数時カンマ区切り） | 該当なしは「—」 |
  | 飛び | 飛びプレイヤー名（複数時カンマ区切り） | 該当なしは「—」 |
  | 焼き鳥 | 焼き鳥プレイヤー名（複数時カンマ区切り） | 該当なしは「—」 |
  | 役満 | プレイヤー名:役名（複数時カンマ区切り） | 該当なしは「—」 |

フッター: ページ番号は必ず CSS カウンタで表示（右下）する。`print.css` に `@page`、`counter()`、`page-break-*` のルールを含めること。

印刷スタイル補足:

- 日本語フォント指定: 日本語が文字化けしないよう、`print.css` に優先フォントとフォールバックを明記する（例: `font-family: "Noto Sans JP", "Noto Serif JP", -apple-system, system-ui, "Segoe UI", Roboto, "Helvetica Neue", Arial, sans-serif;`）。
- 改ページ制御: テーブルや集計ブロックには `break-inside: avoid; page-break-inside: avoid;` を適用して行内分割を避ける。
- 大量データ時のサマリ: 出力件数が多くページ数が増える場合に備え、先頭に「総件数」「想定ページ数（目安）」のサマリブロックを出力する案を追加する（任意）。

## 5.2 フィルタ仕様

成績集計画面の既存フィルタをそのまま受け取り、同条件で PDF を生成する。

1. 日付フィルタ（`filter`）
- `year`: 今年の範囲
- `YYYY-MM-DD`: 指定開催日
- `custom`: 任意期間（`start`, `end`）

2. 試合数フィルタ（`minGames`）
- `""`（空）: 条件なし
- `20`: 20試合以上

3. PDF 出力時は、画面で選択中の `filter/start/end/minGames` を全て引き継ぐ。

## 5.3 クエリパラメータ

印刷ページへは既存の URL パラメータを引き継ぐ。

- `filter`
- `start`
- `end`
- `minGames`

印刷ページ側では [lib/filter-params.ts](lib/filter-params.ts) の `resolveFilterParams` を再利用し、画面と同一条件を解決する。

ヘッダー表示用の条件文言は以下のルールで生成する。

1. `filter=year`: 「対象期間: YYYY-01-01 〜 YYYY-12-31」
2. `filter=YYYY-MM-DD`: 「対象期間: YYYY-MM-DD（開催日）」
3. `filter=custom`: 「対象期間: start 〜 end」
4. `minGames` 空: 「試合数条件: 条件なし」
5. `minGames=20`: 「試合数条件: 20試合以上」

## 7. 受け入れ条件

1. 成績集計画面から PDF 出力導線へ遷移できる。
2. `filter=year`/`filter=開催日`/`filter=custom` の各条件で、A4 横向きの印刷プレビューが表示される。
3. `minGames=条件なし`/`minGames=20試合以上` の各条件で、件数と順位が画面表示と一致する。
4. 成績集計画面の別表欄が、画面と同じ順序で出力に含まれる。
5. 成績集計と対局履歴の両方が出力に含まれる。
6. 行数が多い場合に複数ページへ分割され、内容が欠落しない。
7. 主要ブラウザ（Chrome 系）で日本語が文字化けしない。
8. `npm run build` が成功する。
9. 出力は「4ブロック構成」であること（ヘッダー、成績集計、参考集計＋別表、圧縮対局履歴）。
10. 参考集計（今年の集計（20試合以上））が別ブロックとして出力されること。
11. 対局履歴は圧縮表示で、3人打ちの場合は4列目を非表示にするなど見やすさの調整が適用されること。
12. バッジ等の付帯情報はカンマ区切りで表示されること。

## 8. テスト観点

1. フィルタ組み合わせ
- `year` + 条件なし
- `year` + 20試合以上
- `開催日` + 条件なし
- `開催日` + 20試合以上
- `custom` + 条件なし
- `custom` + 20試合以上

2. データ整合性
- 画面上の対象期間と印刷ページの対象期間が一致
- 成績集計の件数・順位が一致
- 別表の内容と表示順が画面と一致
- 対局履歴の件数が一致

3. 表示ルールと付帯情報
- 3人打ち（3p）の表示ルールが正しく適用される（4列目を隠す等）。
- バッジ/フラグ等はカンマ区切りで表示され、行内の折返し・列幅を考慮して崩れないこと。

4. レイアウト
- A4 landscape でテーブルが横方向に欠けない
- 改ページ時にヘッダーと行の重なりがない

## 9. リスクと対策

1. ブラウザ差による改ページ差分
- 対策: 対応ブラウザを Chrome 系に限定して仕様明記する。

2. データ量増加時の視認性低下
- 対策: 列の優先度を決め、初期版は主要列に絞る。

3. 将来の直接 PDF ダウンロード要望
- 対策: 出力用の表示コンポーネントを分離し、将来 API 生成方式へ移行しやすくする。

## 10. 実装ステップ（次フェーズ）

1. 印刷用ページを追加し、`resolveFilterParams` + `fetchPlayerStats` + `fetchMatchResults` を接続する。
2. 成績集計画面に PDF ボタンを追加し、`filter/start/end/minGames` を保持して印刷ページへ遷移する。
3. 印刷ページヘッダーに対象期間と試合数条件を表示する（システム日付基準は不使用）。
4. 印刷専用 CSS で A4 横向き・改ページ制御を実装する。
5. ビルドと手動確認（印刷プレビュー）を実施する。
6. サーバー側（印刷用サーバーコンポーネント）で「今年の集計（20試合以上）」などの参考集計データを追加で取得し、クライアント描画コンポーネントへ渡す。
7. クライアント側で「4ブロック」構成を描画するコンポーネントを実装する（再利用可能な成績テーブルを流用）。
8. 対局履歴は圧縮表示（行高縮小）でレンダリングし、3人打ち判定時は4列目を非表示にするロジックを組み込む。
9. バッジや付帯情報はサーバーからの配列をカンマ区切りで表示する（長い場合は行折返しの確認）。
10. 印刷トリガ（`window.print()`）はクライアントでロード完了後に一度だけ呼ぶようガード実装（Strict Mode 対策）。
11. 実装後に `npm run build`、手動印刷プレビュー、主要フィルタ組み合わせの確認を実施する。
