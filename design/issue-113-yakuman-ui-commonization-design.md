# Issue #113 設計: 役満登録UI統一とフォーム共通化

- Issue: https://github.com/ytr6310zs-jpg/mj-score-manager/issues/113
- 目的: 対局履歴編集の役満登録方法をスコア入力画面に揃え、将来的なフォーム共通化を進めやすくする。

## 1. 背景

現状は以下の差分がある。

1. スコア入力画面
- 役満登録は「プレイヤー選択 + 役満選択 + 登録」の集中管理型。

2. 対局履歴編集画面
- プレイヤー行ごとに役満登録UIを持つ分散型。

この差分により、操作性の不一致と保守負荷が発生している。

## 2. 今回の実装スコープ

Issue #113 対応として、次を実施する。

1. 対局履歴編集画面の役満登録UIを集中管理型へ変更する。
2. 役満の追加・削除・送信フォーマットをスコア入力画面に揃える。
3. プレイヤー変更や 3p/4p 切替時に、無効な役満エントリを自動除外する。

非スコープ:

- 新規/編集のフォームを完全に1コンポーネント化すること。
- サーバー側の保存スキーマ変更。

## 3. 設計方針

## 3.1 データモデル統一

フロント側の役満一時保持を、スコア入力画面と同等の配列型へ統一する。

- `{ playerName, yakumanCode, yakumanName, points }[]`

編集画面の既存データは初期表示時にこの形式へ変換する。

## 3.2 UI構成統一

編集画面では、プレイヤーごとの役満追加UIを撤去し、次のブロックを単一配置する。

1. 役満対象プレイヤー選択
2. 役満名選択
3. 登録ボタン
4. 登録済み一覧 + 削除ボタン

## 3.3 送信仕様

`yakumanSelections` を JSON 文字列で送る方式を維持する。

- `editMatchAction` 側の `parseYakumanSelections` を変更せず利用可能。

## 3.4 整合性制御

プレイヤー構成が変わった時に以下を行う。

1. 現在有効なプレイヤーに存在しない役満行を削除
2. 選択中プレイヤーが無効になった場合は選択状態をクリア

## 4. 受け入れ条件

1. スコア入力と対局履歴編集で役満登録の操作導線が一致している。
2. 編集画面で役満追加/削除後、保存が成功する。
3. 3p/4p 切替とプレイヤー変更時に不整合な役満データが残らない。
4. `npm run build` が成功する。

## 5. 共通化ロードマップ（次フェーズ）

今回のIssue完了後、段階的に以下を進める。

1. `useMatchFormState`（仮）を導入し、スコア補完・重複検知・飛び/飛ばし整合を共通化
2. `MatchCoreFields`（仮）を導入し、卓種/対局日/プレイヤー/スコア入力UIを共通化
3. 画面差分はコンテナに限定
- 新規: `saveScoreAction`, プレイヤー新規作成導線
- 編集: `editMatchAction`, 既存値ロード、編集完了遷移

この順に進めることで、挙動を維持したままリスクを分割できる。
