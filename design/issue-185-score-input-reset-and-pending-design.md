# Issue #185 設計: スコア入力画面 修正（初期化・大会初期値・保存中リアクション）

- Issue: https://github.com/ytr6310zs-jpg/mj-score-manager/issues/185
- 作成日: 2026-04-28
- 作成者: Copilot

## 1. 目的

スコア入力画面に対して、以下3点をユーザー体験として明確化・安定化する。

1. 登録後に入力フォームを初期化する。
2. 大会を前回入力値で初期セットする。
3. 保存中であることを明確に表示し、二重送信を防ぐ。

Issue コメントで指定された Iteration 3 (P1) の位置づけに従い、Issue #105 のテスト基盤を活用して自動回帰確認まで含める。

## 2. 現状整理

対象実装の現状。

1. スコア入力UIは `components/score-form.tsx` で `useActionState(saveScoreAction, initialState)` を利用している。
2. 保存中は送信ボタンが `disabled` になり、ラベルは `保存中...` に切り替わる。
3. 大会初期値は `lib/tournament-preference.ts` の localStorage から復元している。
4. 保存成功後は `resetRoundFields()` が呼ばれるが、プレイヤー選択・卓種・対局日などが完全には初期化されない。

要求との差分。

1. 「登録後に画面リロード等で初期化」に対し、現状は部分リセットのみ。
2. 大会初期セットは概ね実装済みだが、保存フローに紐づく仕様が明文化されていない。
3. 保存中表示は最小限で、連打耐性やフォーム全体の操作抑止が弱い。

## 3. 要件定義（Issue #185の具体化）

### 3.1 機能要件

1. 保存成功時に、次局入力開始に必要な初期状態へフォームを戻す。
2. 大会初期値は「最後に使った大会ID」を優先し、無効値時はフォールバックする。
3. 保存処理中はユーザーが「処理中」と分かる表示を行う。
4. 保存処理中および送信直後の短時間連打で二重送信されない。

### 3.2 テスト要件（Issueコメント反映）

1. 入力修正の主要シナリオに対するE2Eを追加（登録後状態/初期化/保存中リアクション）。
2. 壊れやすい箇所にUnit/Integrationテストを追加。
3. 失敗ケース（不正入力または二重送信）を最低1件自動化。
4. `npm run build` と `npm test` を成功させる。

## 4. スコープ

### 4.1 対象

1. `components/score-form.tsx` の状態初期化と送信中UI改善。
2. 大会初期値復元ロジックの仕様明文化と必要な微修正。
3. E2E/Unit/Integration テスト追加（Issue #105基盤を利用）。

### 4.2 非対象

3. 新たな認証・認可要件の追加。

## 5. 設計方針

### 5.1 保存成功後の初期化方針

採用: フルリロードではなく、クライアント状態を明示的に初期化する。

理由:

1. 体感速度を維持できる。
2. 既存の `useActionState` パターンと整合する。
3. 「リロード等」の要件を、より制御可能な形で満たせる。

初期化対象:

1. 卓種: 既定値へ戻す（既定は現状準拠）。
2. プレイヤー選択: 全スロット空に戻す。
3. スコア入力: クリア。
4. 飛び/飛ばし/焼き鳥/役満/メモ: クリア。
5. 対局日: 当日デフォルトへ戻す。
6. 大会: 「最後に使った大会ID」を再適用（保存した大会を優先）。

### 5.2 大会初期セット方針

1. 保存成功時に、送信時大会IDを `setLastTournamentId` で確実に更新する。
2. 初回表示時は `getLastTournamentId()` を参照し、有効IDなら採用する。
3. 無効/未保存時は大会一覧先頭へフォールバックする。

備考:

1. 現在の実装方針を維持しつつ、保存成功フックでの更新を明示化する。

### 5.3 保存中リアクションと二重送信防止

1. 既存の `pending` 表示（ボタン文言/disabled）を維持。
2. 送信可能な主要入力領域を `pending` 中に操作不可にする（`fieldset disabled` など）。
3. 連打対策として、`pending` 反映前の短時間窓でも2回目送信を拒否するクライアントガードを追加する。

制約:

1. 本IssueではDBレイヤの厳密な冪等性トークンは導入しない。

3. 失敗ケースとして不正入力（スコア合計不整合）または二重送信抑止を自動化。

## 8. リスクと対策

1. リスク: 初期化し過ぎて連続入力の利便性が下がる。
   対策: 大会のみ前回値を維持し、他項目を初期化する。
2. リスク: 二重送信防止の実装で通常送信まで阻害する。
   対策: pending遷移とエラー復帰時のフラグ解除条件をテストで固定化する。
3. リスク: E2Eが非同期タイミングで不安定になる。
   対策: 送信ボタン状態と成功メッセージを明示的に待機し、検証ポイントを限定する。

## 9. 受け入れ基準（設計時点）
