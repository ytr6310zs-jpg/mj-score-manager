## コンテキスト

Issue #119: 「成績集計 各項目の1～3位が分かるとよい」。
現在は行全体の背景色で順位（総合順位）を示しているが、各指標ごとの1〜3位が分かるように、
セル（個々の数値）単位で上位3名を可視化する要望。

また、指標によって「高い方が良い（降順）」と「低い方が良い（昇順）」が混在するため、
各指標に対して向き（direction）を明示して評価する必要がある。

## 問題

- 行全体の背景色だと、どの指標で上位なのかが分かりにくい。
- 指標ごとに並び替え方向が異なるため、単純に“上位3名”を取るだけでは誤認が発生する。
- 表示スペース（特にモバイル）やアクセシビリティにも配慮する必要がある。

## 決定

1. PC のテーブル表示では、各数値セルに対してその指標の上位3名（1/2/3位）を色付けして可視化する。
   - 色は `1位: gold` / `2位: silver` / `3位: bronze` のアクセシブルな調整されたカラーパレットを使用。
2. モバイルカード表示では、各指標の値の横に小さなバッジ（1/2/3）を表示し、スペースが厳しい箇所はアイコンで代替する。
3. 指標ごとの評価向き（降順 / 昇順）は設定で管理し、同一ロジックを PC/モバイル双方で利用する。
4. 同点の扱い: 3位に到達する値と同値のプレイヤーは同じ順位としてハイライトする（例: 3位値と同じ値は全員3位表示）。
5. 実装はクライアント側（`components/stats-sortable-table.tsx`）で行い、レンダリング毎に `stats` を一度だけ分析して上位セットを算出する。

## 指標と初期評価向き（提案）

--- 降順（数値が大きいほど上位） ---
- `games`（対局数）
- `topCount`（トップ回数）
- `topRate`（トップ率）
- `secondRate`（2位率）
- `thirdRate`（3位率）
- `lastAvoidanceRate`（ラス回避率）
- `tobashiCount`（飛ばし）
- `yakumanCount`（役満）
- `tobashiRate`（飛ばし率）
- `tobiAvoidanceRate`（飛び回避率）
- `yakitoriAvoidanceRate`（焼き鳥回避率）
- `setaiRate`（接待率）

--- 昇順（数値が小さいほど上位） ---
- `lastCount`（ラス回数） ※ ラス回数は低い方が良いので昇順評価

その他の指標はプロジェクト設定で明示的に追加できるようにする。

（注）上記向きは Issue 本文の記載を踏まえた提案。最終決定はチームでの確認を推奨。

## 実装方針（詳細）

1. 指標向き設定
   - `components/stats-sortable-table.tsx` に `METRIC_DIRECTION: Record<string, 'asc'|'desc'>` を用意し、評価向きを定義する。

2. 上位3名の算出
   - `stats`（PlayerStats[]）を受け取り、対象指標ごとに次を行う。
     - 各プレイヤーの値を取得してソート（`asc`/`desc` に応じて昇降順）。
     - トップ3の“閾値”を決定し（1位値、2位値、3位値）、その閾値と同値のプレイヤーは同順位として扱う。
     - 出力は `Record<metric, { first: Set<name>, second: Set<name>, third: Set<name> }>` のようなマップ。
   - 計算は `useMemo`（またはサーバー側でも可能）で行い、`stats` が変わったときのみ再計算する。

3. テーブル描画
   - 各セルを描画する際に、上記マップを参照して `1位/2位/3位` いずれかに該当すれば対応する CSS クラス（背景色 + 文字色）を付与する。
   - ツールチップや `aria-label` に順位情報を含め、アクセシビリティを確保する。

4. モバイルカード描画
   - モバイルカード内の各指標の値横に小さなバッジ（色付きまたは数字）を表示。
   - スペース不足時はカラードット＋`title`属性で順位を示す。

5. パフォーマンス
   - プレイヤー数は通常それほど大きくない前提だが、指標数 × プレイヤー数の計算量は `O(M*N)` になる。
   - 実行コストを抑えるため `METRICS_TO_HIGHLIGHT` を限定リストにして必要な指標のみ計算する。

6. テスト
   - ランキング計算のユニットテストを追加（同点処理、昇降順反転、混在データを検証）。

## 受け入れ基準

- PC テーブルで任意の指標に対して該当するプレイヤーのセルが 1/2/3 位として色分けされていること。
- モバイルカードで同様の順位バッジが表示されること（スペースが厳しい場合はアイコン表示）。
- 指標ごとの昇降順設定に従って上位が選定されていること（設定ファイルで調整可能）。
- 同点のプレイヤーは同順位でハイライトされること。

## 影響範囲

- 変更: `components/stats-sortable-table.tsx`, `app/stats/page.tsx`（モバイルカード描画）
- 追加: `design/adr/0014-issue-119-cell-rank-highlighting.md`（このファイル）、テストファイル

## 代替案

1. サーバー側で事前に各指標の上位セットを算出して返す
   - クライアントの計算負荷を減らせるが、サーバー API へ変更が必要で実装コストが増える。
2. ヘッダで順位インジケータを表示（セルでの色付けは行わない）
   - UI がシンプルになるが、視認性が下がる。

## 実施ステップ（推奨）

1. `design/adr/0014-issue-119-cell-rank-highlighting.md` をコミット（このファイル）。
2. `components/stats-sortable-table.tsx` に `METRIC_DIRECTION` / `METRICS_TO_HIGHLIGHT` を追加し、上位セット算出関数を実装。
3. テーブルのセルにハイライトクラスを適用、`aria`/tooltip を追加。
4. `app/stats/page.tsx` のモバイルカードにバッジ表示を追加。
5. ユニットテストを追加して `npm run build` を実行、表示を確認。
6. コミット・PR 更新。
