## コンテキスト

Issue #115 では、対局履歴で「飛び、焼き鳥、飛ばしが1つしか表示されない」不具合の修正が求められている。

現状の `app/matches/page.tsx` では、プレイヤーごとの特記事項表示に `resultBadge()` を利用しているが、この関数は単一の文字列を返す実装になっている。

```ts
function resultBadge(player: MatchPlayer): string {
  if (player.isTobi) return "飛び";
  if (player.isTobashi) return "飛ばし";
  if (player.isYakitori) return "焼き鳥";
  return "";
}
```

このため、1人のプレイヤーに複数のフラグが立っていても、最初に一致した1件しか表示されない。

また、Issue #116 の設計で、対局単位の集約表示を削除し、飛び/飛ばし/焼き鳥はプレイヤー行バッジへ集約する方針を採用している。そのため、各プレイヤー行で複数の特記事項を正しく並列表示できることが前提になる。

## 問題

- `isTobi` / `isTobashi` / `isYakitori` が同時に真でも、1件しか表示されない
- Issue #116 の省スペース化後は、プレイヤー行バッジが唯一の表示箇所になるため、この欠落がそのまま情報欠損になる
- モバイル/PCで同じロジックを使っているため、両方で同じ不具合が発生する

## 決定

1. 単一文字列ではなく、複数バッジを返す構造に変更する。
   - `resultBadge()` は廃止または置き換えとし、`player` から表示対象のラベル配列を生成する。
   - 返却型は `string[]` を基本とする。

2. バッジはフラグごとに独立して表示する。
   - `isTobi` が真なら `飛び`
   - `isTobashi` が真なら `飛ばし`
   - `isYakitori` が真なら `焼き鳥`
   - 条件が複数真なら、該当する全バッジを表示する

3. 表示順は固定する。
   - 並び順は `飛び` → `飛ばし` → `焼き鳥` とする。
   - モバイル/PCで同じ順序を使い、表示の一貫性を保つ。

## 実装方針（詳細）

1. バッジ生成処理の置き換え
   - `app/matches/page.tsx` のローカル関数を、配列を返す `resultBadges(player)` 相当へ変更する。
   - 関数内ではフラグごとに配列へ push し、最終的に配列を返す。

2. モバイル表示の更新
   - 現在は1つの文字列を `<p>` で表示しているため、複数バッジを横並びまたは折り返し可能なインライン要素群へ変更する。
   - 役満バッジとの視覚的優先度は維持しつつ、特記事項バッジが複数でも崩れないようにする。

3. PC表示の更新
   - 現在は1つの `<span>` で表示しているため、複数バッジを並べて描画する構造に変更する。
   - 1行内でバッジ数が増えても読めるよう、`flex` と `flex-wrap` を使う。

4. 既存設計との整合
   - Issue #116 の「対局単位の集約表示を削除し、プレイヤー行へ集約する」方針は維持する。
   - 今回の変更は、その集約先で情報を欠落させないための補完と位置づける。

## 受け入れ基準

- `isTobi` と `isYakitori` が同時に真のプレイヤーで、両方のバッジが表示される。
- `isTobi` と `isTobashi` と `isYakitori` が同時に真のプレイヤーで、3つすべてのバッジが表示される。
- モバイル/PCの両表示で、複数バッジがレイアウト崩れなく表示される。
- 既存の役満バッジ表示や順位表示に影響しない。

## 影響範囲

- 変更: `app/matches/page.tsx`
- 非変更: `lib/matches.ts`, DB スキーマ, API, 保存データ形式

## 代替案

1. 単一バッジのまま優先順位だけ見直す。
   - 実装は小さいが、複数情報を同時に見せられず問題を解決しないため不採用。

2. 特記事項を1つの連結文字列（例: `飛び / 焼き鳥`）で表示する。
   - 実装は可能だが、既存のバッジ表現と一貫せず視認性が下がるため不採用。

## テスト / 検証

- 手動確認
  - 複数フラグを持つプレイヤーの表示で全バッジが見えること
  - モバイル/PCで折り返しや余白が破綻しないこと
- 自動確認
  - `npm run build` を実行し、型・ビルドエラーがないことを確認する

## リスク

- バッジ数が増えることで1行の横幅が不足する可能性がある
- 特にPC表でプレイヤー名、スコア、役満バッジと並ぶため、折り返し制御が不十分だと可読性が下がる

## 実施ステップ

1. `app/matches/page.tsx` の特記事項バッジ生成を配列ベースへ変更する。
2. モバイル/PCそれぞれの描画を複数バッジ対応に変更する。
3. `npm run build` を実行して品質確認。
4. 複数バッジ表示を手動確認する。
