## コンテキスト

Issue #112 では、画面遷移時に「遷移が始まっている」ことがユーザーに伝わるリアクションの追加を求められている。

現状はヘッダーのメニュー押下後、遷移完了まで視覚的な変化が少なく、通信やサーバー処理が遅いケースで「押せていないのか、遷移中なのか」が分かりづらい。

## 問題

- ヘッダーの遷移操作（`/`, `/matches`, `/stats`）で待機中のフィードバックが弱い
- Next.js App Router の遷移待機時に、全体としてローディング状態を示す共通UIがない
- UI 応答がないように感じると、重複クリックや操作離脱につながる

## 決定

1. 画面上部に、遷移中を示すグローバルインジケーターを表示する。
2. ヘッダーの遷移ボタンに Pending 状態を導入し、押下直後に見た目を変える。
3. 実装は UI 層のみとし、API/DB スキーマは変更しない。

## 実装方針（詳細）

1. `app/loading.tsx` を追加する。
   - App Router の遷移待機中に表示される共通ローディングUIとして扱う。
   - 最小構成は以下。
     - ページ上部の細いプログレスバー（アニメーション）
     - `aria-live="polite"` で「読み込み中」を支援技術に通知

2. `components/app-header.tsx` の主要遷移ボタンに Pending 表示を入れる。
   - クリック時に「選択中 / 移動中」を示すクラス（例: opacity, cursor）を適用する。
   - 二重押下防止のため、遷移開始後は同一ボタンの連打を抑制する。
   - 見た目のみでなく `aria-busy` 相当の属性で状態を伝える。

3. 表示ルールを統一する。
   - 速い遷移（体感 200ms 未満）は瞬間表示で問題ない。
   - 遅い遷移では上部インジケーターとボタン状態が両方見える。
   - 現在ページ判定（`current`）による default/outline の既存仕様は維持する。

## 受け入れ基準

- ヘッダーの遷移操作後、即座に視覚的リアクションが出る。
- 遷移待機が発生するケースで、画面上部インジケーターが表示される。
- キーボード操作・スクリーンリーダー利用時に、遷移中状態が把握できる。
- 既存のページ遷移先と active 表示ロジックが壊れない。

## 影響範囲

- 追加: `app/loading.tsx`
- 変更: `components/app-header.tsx`
- 非変更: DB, API, 認証ロジック

## 代替案

1. `loading.tsx` のみ導入する。
   - 実装は容易だが、押下直後の局所的フィードバックが弱い。

2. ヘッダーのボタン状態変更のみ導入する。
   - 押下直後は分かりやすいが、全体遷移の待機を一貫して示せない。

本 ADR では、ユーザー体感を優先して両方を併用する。

## テスト / 検証

- 手動確認
  - 3つの主要遷移（`/`, `/matches`, `/stats`）で押下直後の変化が見えること
  - 通信遅延を再現した場合でも、遷移中であることが判別できること
- 自動テスト
  - 既存テストへの直接影響は軽微
  - 必要に応じて E2E で「遷移トリガー後にインジケーターが現れる」確認を追加

## リスク

- インジケーターの点滅が短すぎると視認できない可能性がある。
- 逆に長く残ると完了後の違和感につながるため、表示タイミング調整が必要。

## 実施ステップ

1. `app/loading.tsx` を追加してグローバル遷移インジケーターを実装。
2. `components/app-header.tsx` に Pending 表示を追加。
3. `npm run build` で型/ビルド確認。
4. 主要導線を手動確認し、必要なら文言・表示時間を微調整。
