## コンテキスト

Issue #118 では「成績集計 PCレイアウト時にソート機能」の追加が求められている。

現状の `app/stats/page.tsx` はサーバーで集計済みデータを受け取り、PC レイアウト（`md:block` のテーブル）に固定順で表示している。表示順は `lib/stats.ts` 側で `totalScore` 降順に確定されており、ユーザーが列単位で並び替えることはできない。

そのため、特定指標（例: 飛ばし率、役満回数、対局数）で上位プレイヤーを確認したい場合に、手作業で目視比較する必要がある。

## 問題

- PC テーブルで列ごとの比較がしづらい
- デフォルトの総合順（合計得点降順）しか見られない
- 集計指標が増えるほど、特定観点の分析コストが上がる
- モバイルカード表示は情報密度と操作性の都合上、同等のソート UI をそのまま適用しづらい

## 決定

1. PC レイアウトの成績テーブルに列ソート機能を追加する。
   - 対象は `md` 以上のテーブル表示のみとする。
   - モバイルカード表示は今回のスコープ外とする。

2. ソートはクライアント側で実行する。
   - 集計データ取得は従来どおりサーバー側で実施する。
   - 表示順のみクライアントコンポーネントで並び替える。

3. ソート対象列を明示する。
   - 対象: `名前`, `合計`, `順位`, `対局数`, `トップ回数`, `ラス回数`, `トップ率`, `ラス回避`, `飛ばし`, `飛び`, `焼き鳥`, `役満`, `飛ばし率`, `飛び回避率`, `焼き鳥回避率`, `接待率`
   - 各ヘッダー押下で `昇順 -> 降順 -> 昇順 ...` を切り替える。

4. 初期表示は既存仕様を維持する。
   - 初期ソートは `合計` の降順とし、現行の表示順と一致させる。
   - `順位` は「総合順位（合計得点基準）」として固定値を表示し、他列ソート時にも再計算しない。

5. ヘッダーにソート状態を表示する。
   - アクティブ列と方向（昇順/降順）が視覚的に分かる表示を追加する。
   - ヘッダー操作はキーボードでも実行可能な要素（`button`）で提供する。

## 実装方針（詳細）

1. テーブル表示をクライアントコンポーネントへ分離
   - 新規 `components/stats-sortable-table.tsx` を作成する。
   - `app/stats/page.tsx` から `stats` を props で渡して描画する。

2. ソート状態の管理
   - `SortKey`（列識別子）と `SortDirection`（`asc` / `desc`）を state で保持する。
   - 初期値は `SortKey = totalScore`, `SortDirection = desc`。

3. 並び替え処理
   - `useMemo` でソート済み配列を算出する。
   - 数値列は数値比較、名前列は `localeCompare` を用いる。
   - 同値時は元インデックスを用いた安定ソートで行順の不自然な入れ替わりを防ぐ。

4. ヘッダー UI
   - ソート可能列は `<th>` 内に `<button>` を配置する。
   - クリックで対象列を選択し、同一列再クリックで方向を反転する。
   - ソート方向インジケータ（例: ▲/▼）を列名横に表示する。

5. 既存見た目との整合
   - 行背景色（順位ベース）・バッジ・数値右寄せなど既存スタイルを維持する。
   - Sticky な名前列レイアウトを維持し、横スクロール挙動を壊さない。

## 受け入れ基準

- PC レイアウトで任意の対象列を押下すると並び順が切り替わる。
- 同一列を連続押下したとき、昇順/降順がトグルする。
- 初期表示が従来どおり「合計降順」である。
- 他列ソート時にも `順位` 表示は総合順位のまま変化しない。
- モバイル表示（カード一覧）の見た目・挙動に変更がない。
- `npm run build` が成功する。

## 影響範囲

- 変更: `app/stats/page.tsx`
- 追加: `components/stats-sortable-table.tsx`
- 非変更: `lib/stats.ts`, DB スキーマ, API, 保存データ形式

## 代替案

1. サーバーサイドソート（クエリパラメータ連動）
   - URL 共有性は高いが、実装量が増え、列クリックごとに再フェッチが必要になるため今回は不採用。

2. 合計列のみソート可能に限定
   - 実装は小さいが、Issue の「成績集計での比較操作性向上」を十分に満たさないため不採用。

## テスト / 検証

- 手動確認
  - PC で各列ソートが動作すること
  - 列方向インジケータが現在状態と一致すること
  - モバイル表示に回帰がないこと
- 自動確認
  - `npm run build` を実行し、型・ビルドエラーがないこと

## リスク

- クライアント側並び替えロジックの重複により、将来 `lib/stats.ts` の項目追加時に追従漏れが起こる可能性がある
- Sticky 列とヘッダー button の組み合わせで、ブラウザ差異による表示崩れが発生する可能性がある

## 実施ステップ

1. `components/stats-sortable-table.tsx` を追加し、ソート状態管理とテーブル描画を実装する。
2. `app/stats/page.tsx` の PC テーブル描画を新コンポーネント呼び出しへ置き換える。
3. `npm run build` で型/ビルド検証を実施する。
4. PC/モバイルで手動確認する。
